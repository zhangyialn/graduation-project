# 公务用车在线申请与审批系统（答辩版）

本项目面向高校/企事业单位公务用车场景，目标是将传统线下“纸质申请 + 人工审批 + 电话调度”的流程，升级为可追踪、可审计、可统计的线上闭环系统。

## 一、课题背景与痛点

传统公务用车管理普遍存在以下问题：
- 申请流转慢：纸质单据或群聊沟通，审批链路不透明。
- 资源冲突多：车辆/司机占用信息分散，容易重复调度。
- 过程留痕弱：行程节点、里程油耗、审批责任难以追溯。
- 统计滞后：报表依赖人工汇总，无法实时反映部门与车辆使用情况。

## 二、系统建设目标

- 实现用车申请、审批、调度、行程、费用、报表的全流程线上化。
- 提供角色化权限控制，保证不同岗位“看得见、做得对、留得住”。
- 通过规则校验和事务一致性控制，降低流程错单与数据不一致风险。
- 提供可视化统计能力，为管理决策提供数据支撑。

## 三、系统角色与职责

- 普通用户（user）：发起申请、查看本人申请与行程、对行程评分。
- 司机（driver）：查看任务、确认接客、填报里程油耗、维护司机/车辆状态。
- 审批员（approver）：审批申请、参与调度、查看审批与运行统计。
- 管理员（admin）：用户/部门/车辆治理、管理员初始化与系统配置。

## 四、核心业务闭环

1. 用户提交用车申请。
2. 审批员/管理员执行审批（通过或驳回）。
3. 对通过申请创建调度并启动任务。
4. 司机执行任务，确认接客并提交行程数据。
5. 乘客结束行程，系统完成费用归档。
6. 报表按部门、车辆、司机、用户维度输出统计结果。

## 五、关键功能亮点

### 1. 全链路状态驱动
- 申请、调度、行程均采用明确状态流转，前后端同口径展示。
- 对关键状态变更设置业务校验，避免越权与脏数据写入。

### 2. 调度与资源冲突控制
- 调度创建时校验司机、车辆和时段占用冲突。
- 司机/车辆在进行中任务期间限制不合理状态变更。

### 3. 行程与费用闭环
- 司机可填报里程与油耗，乘客确认完成后统一归档。
- 油价能力支持区域油价读取与批量入库，支撑费用口径稳定。

### 4. 账号与安全能力
- JWT 鉴权 + 角色授权，前后端统一权限边界。
- 首个管理员初始化流程，保障系统首次部署安全可控。
- 支持找回密码与账号设置，完善日常运维体验。

### 5. 可视化管理支持
- 输出部门用车、费用、车辆使用、司机工作量、用户画像等统计结果。
- 为“预算管理、资源配置、流程优化”提供量化依据。

## 六、技术路线与架构概览

- 前端：Vue 3 + Pinia + Element Plus + ECharts + Vite。
- 后端：Flask + SQLAlchemy + JWT + Bcrypt。
- 数据库：MySQL。
- 架构：路由层（接口装配） + 控制器层（编排） + 服务层（复杂业务下沉） + 模型层（数据定义）。

## 七、重构成果（工程化提升）

本阶段对系统进行了结构重构，重点包括：
- 将审批流、调度流程、司机工作台、行程结束、导入等复杂逻辑下沉到服务层。
- 控制器保留接口编排职责，降低单文件复杂度，提升可维护性。
- 引入统一事务装饰器，标准化写接口的 commit/rollback 与业务异常映射。
- 清理无效接口与无用代码，减少历史包袱和潜在故障点。

## 八、项目价值

- 管理价值：提升审批与调度效率，降低人工协同成本。
- 风险价值：关键节点可追溯，减少错派车、重复派车、数据不一致。
- 数据价值：形成结构化业务数据资产，支撑精细化管理与持续优化。

## 九、后续可扩展方向

- 引入自动化测试（单测 + 接口回归）和 CI/CD 流程。
- 增加审计日志与异常告警面板。
- 对接地图与轨迹能力，增强运行过程可视化。
- 引入移动端适配，满足司机与审批移动办公场景。
